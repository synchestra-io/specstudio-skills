## Problem Statement

How might we let SpecStudio adopters bootstrap a SpecScore-managed project — spec/ tree, integration instruction snippet, project config — in one conversational step instead of relying on lazy in-skill bootstrapping scattered across ideate and specify?

## Context

This Idea was scaffolded alongside (and immediately after) third-party-skill-integration-contracts on 2026-05-08, in response to the observation that SpecStudio's current bootstrap is scattered: `specstudio:ideate` lazily creates `spec/ideas/` on first use, `specstudio:specify` lazily creates `spec/features/`, and the proposed Producer-shape instruction snippet has no install path. Two further pressures emerged: (1) a project-level config file (proposed name `specscore.yaml` or `.synchestra/config.yaml`) is one of the candidate mechanisms for reviewer registration in the integration contract, and that file has no canonical creation path either; (2) bootstrapping spans multiple platform-specific agent-instructions files (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`, `.cursor/rules/`) and no skill currently knows how to detect and target the right one. A unified init step solves all three.

## Recommended Direction

Build a `specstudio:init` skill that runs a short interactive wizard to bootstrap a SpecScore-managed project. The skill prefers the deterministic CLIs underneath — `specscore init` for the spec tree and `specscore.yaml` config (per the canonical [SpecScore Repo Config Feature](https://github.com/synchestra-io/specscore/blob/main/spec/features/repo-config/README.md)), and `synchestra init` for orchestration-side setup (state-repo pointers, runtime caches, any `.gitignore` rules Synchestra needs) — and falls back to the AI agent performing the file writes directly only when a CLI is not on PATH. When a required CLI is missing, the skill delegates installation to the dedicated install skills — `specscore:install` for specscore, `synchestra:install` for synchestra — rather than running install commands itself. Init detects "CLI not on PATH," asks the user for consent to invoke the install skill, hands off, and resumes once the install skill returns. Install URLs, supported methods (canonical `curl -fsSL https://specscore.md/get-cli | sh`, plus `brew`, `npm`, manual), signature verification, and version pinning all live in the install skills, not in init. The canonical install instructions live at `https://specscore.md/install`. Both install skills are hard prerequisites for this Feature — `specscore:install` already exists; `synchestra:install` must ship (or its existence be confirmed) before `specstudio:init` can be promoted to a Feature.

The wizard asks a small number of focused questions (which platform agent-instructions file to update, whether to scaffold optional dirs like `spec/research/`, whether to register a custom viewer, which orchestration features to enable), then idempotently scaffolds: (1) the `spec/` tree with lint-clean indexes via `specscore init`; (2) `specscore.yaml` with a default `project:` block (line 1 is the mandatory schema-pointer comment per the canonical schema); (3) the canonical Producer-shape instruction snippet — defined by the [third-party-skill-integration-contracts](third-party-skill-integration-contracts.md) Feature — appended to the right platform agent-instructions file with an opt-in confirmation; (4) any orchestration-side artifacts via `synchestra init`. Idempotence is mandatory — running the skill on an already-initialized project reconciles drift via direct repo inspection, never clobbers user edits, and never relies on a hidden state file.

`specstudio:init` becomes the canonical bootstrap path. The existing lazy-bootstrap behavior in `specstudio:ideate` and `specstudio:specify` is rewritten to delegate to `specstudio:init` (or the `specscore init` CLI directly) rather than duplicate the bootstrap logic, removing the scattered-bootstrap problem entirely. After successful initialization the skill emits a `project.initialized` event for Synchestra Hub to consume.

**Platform detection rule for the snippet install step.** When the repo has only `CLAUDE.md`, target it. When the repo has only `AGENTS.md` (or `GEMINI.md`, or another single platform file), target that. When the repo has both `CLAUDE.md` and `AGENTS.md`, the skill MUST check whether `CLAUDE.md` references `AGENTS.md` (e.g., a "see AGENTS.md" pointer or an `@AGENTS.md` import) — if it does, target only `AGENTS.md` (CLAUDE.md is acting as a thin redirect). Otherwise ask the user which to update or whether to update both. The same rule generalizes to other multi-platform combinations.

## Alternatives Considered

- **Keep duplicated lazy in-skill bootstrap, no dedicated init.** Each skill (`ideate`, `specify`, future skills) independently creates the `spec/` subdirectory it needs on first invocation. Lost because it doesn't solve the snippet install problem (no skill currently writes to `CLAUDE.md` / `AGENTS.md`), it gives no canonical home for `specscore.yaml` creation, leaves the `synchestra init` orchestration-side setup orphaned, and "scattered bootstrap" is precisely the friction this Idea exists to remove. The replacement: `specstudio:ideate` and `specstudio:specify` continue to handle first-use cases, but they delegate to `specstudio:init` (or directly to the `specscore init` CLI) rather than duplicating the bootstrap logic. One canonical implementation, multiple invocation surfaces.
- **CLI-only `specscore init` with no conversational skill.** A single `specscore init` command does everything deterministically with flags. Lost because the bootstrap step that needs human judgment (which platform agent-instructions file to target, whether to overwrite an existing snippet, whether to opt into an optional config) is exactly the surface a conversational skill handles well. The CLI subcommand exists underneath as the deterministic engine; the skill is the conversational wrapper that gathers preferences before invoking it.
- **Bundle init into the third-party-integration Feature itself.** Make installing the snippet a sub-requirement of the integration contract Feature. Lost because installation is a separate concern from the contract — the contract defines what the snippet says; installation defines how a project gets bootstrapped end-to-end (which involves more than just the snippet). Bundling violates single-responsibility and would make the integration Feature's scope much wider than its name suggests.
- **Adopt an existing project-init tool (Yeoman, Cookiecutter, Plop, Copier) as the engine.** Lost because it adds an external runtime dependency (Node, Python, etc.) for a job that's mostly file scaffolding plus one or two interactive prompts — the `specscore` CLI already exists and is the natural home. Adopters who want to template SpecStudio-managed projects from a central source can build a Cookiecutter template that wraps `specscore init`; that's outside MVP scope.

## MVP Scope

A working `specstudio:init` skill that orchestrates a short wizard, prefers `specscore init` and `synchestra init` CLIs, and falls back to direct AI-agent file writes when a CLI is not on PATH. When run in an empty git-tracked directory, it produces: (a) lint-clean `spec/ideas/README.md` and `spec/features/README.md` via `specscore init`; (b) a `specscore.yaml` with the mandatory schema-pointer header and a default `project:` block; (c) the canonical instruction snippet appended to the right platform agent-instructions file per the platform-detection rule (`CLAUDE.md` / `AGENTS.md` / `GEMINI.md`); (d) any orchestration-side artifacts via `synchestra init`; (e) a `project.initialized` event emitted on success. Optional CLI install via package manager — gated behind explicit user consent — covers the "specscore not on PATH" case. Idempotent on rerun via direct repo inspection (no state file). The companion `specscore init` CLI subcommand is a prerequisite (owned by the specscore repo, not this Feature), as is `synchestra init` (owned by the synchestra runtime). Out of MVP: no GUI, no plugin-manager auto-install of the platform itself (Claude Code, Codex, etc.), no migration of pre-existing layouts. One `specstudio:specify` session producing one Feature — `spec/features/skills/init/`.

## Not Doing (and Why)

- GUI installer or web wizard — CLI and conversational skill only.
- Multi-repo / monorepo orchestration — single-repo init in MVP; multi-repo is a separate concern.
- Plugin-manager / platform-installer integration (e.g. auto-installing the SpecStudio plugin into Claude Code, Codex, or Gemini CLI) — adopters bring their own platform; init only configures the repo. Note: this is distinct from installing the `specscore` and `synchestra` CLIs themselves, which IS in scope (gated behind user consent, via brew/npm/etc.).
- `.gitignore` modifications — owned by `synchestra init`. If Synchestra runtime caches need ignoring (e.g. a `.synchestra/cache/` directory), `synchestra init` writes the rule, not `specstudio:init` directly.
- Migration from existing `docs/`-based or otherwise-shaped spec layouts — out of scope for this Idea, but explicitly worth its own future Idea: `specstudio:migrate` (or similar) once enough adopters have non-canonical pre-existing layouts to justify a dedicated skill. Track demand; do not preemptively build.
- Authoring the canonical instruction snippet itself — owned by [`third-party-skill-integration-contracts`](third-party-skill-integration-contracts.md); `specstudio:init` consumes the snippet, doesn't define it.
- Authoring the schema for `specscore.yaml` — owned by the [SpecScore Repo Config Feature](https://github.com/synchestra-io/specscore/blob/main/spec/features/repo-config/README.md). `specstudio:init` writes a file conforming to that schema; it doesn't define the schema.

## SpecScore Integration

- **New Features this would create:** `spec/features/skills/init/` — the `specstudio:init` skill Feature itself (sibling to `skills/ideate/` and `skills/specify/`). The accompanying `specscore init` CLI subcommand is owned by the [specscore CLI repo](https://github.com/synchestra-io/specscore), not this Feature. The `synchestra init` CLI subcommand is owned by the synchestra runtime repo, not this Feature.
- **Existing Features affected:** [`spec/features/skills/ideate/`](../features/skills/ideate/README.md) and [`spec/features/skills/specify/`](../features/skills/specify/README.md) — both currently bootstrap their respective `spec/` subdirectories lazily via direct file writes. After this Feature ships, both skills are rewritten to delegate to `specstudio:init` (or invoke `specscore init` directly) rather than duplicating bootstrap logic. The behavior change is transparent to users (lazy-bootstrap on first use still happens) but the implementation moves to one canonical place. The existing [`specscore:install`](https://github.com/synchestra-io/specscore) skill is invoked by `specstudio:init` when `specscore` is not on PATH — no changes to `specscore:install` itself, but its handoff contract becomes a load-bearing dependency. Symmetrically, the `synchestra:install` skill is invoked when `synchestra` is not on PATH; same contract dependency. The `specstudio:plan` Idea is unaffected — plan operates on existing Features and creates no new spec subdirectories.
- **Hard prerequisites:** Both install skills MUST exist before this Feature can be specified. `specscore:install` already exists. `synchestra:install` existence MUST be confirmed at specify time; if absent, an Idea for `synchestra:install` (mirroring the `specscore:install` shape) MUST be scaffolded and approved before this Idea promotes to a Feature.
- **Dependencies:** [`spec/ideas/third-party-skill-integration-contracts.md`](third-party-skill-integration-contracts.md) — this Idea depends on the integration contract Idea being at minimum `Approved` (preferably `Specified`) before this one can be promoted to a Feature, because the canonical Producer-shape instruction snippet is what init installs. The two Ideas can be ideated and approved in parallel, but the init Feature should not be specified before the integration contract Feature has at least defined the snippet artifact. Additional upstream dependency: the [SpecScore Repo Config Feature](https://github.com/synchestra-io/specscore/blob/main/spec/features/repo-config/README.md) defines the `specscore.yaml` schema this skill writes; init conforms to it but does not own it. Sister dependency: the `synchestra init` CLI subcommand contract — the orchestration-side init is the synchestra runtime's job, not this Feature's; this Feature only specifies how `specstudio:init` invokes it.

## Outstanding Questions

- **CLI installation is delegated symmetrically.** When `specscore` is not on PATH, the wizard MUST invoke the existing [`specscore:install`](https://github.com/synchestra-io/specscore) skill. When `synchestra` is not on PATH, the wizard MUST invoke `synchestra:install` (same shape, same handoff protocol). Init owns "is the CLI present?" detection and consent-to-invoke-install; install skills own URLs, methods (curl-pipe-sh, brew, npm, manual), signature verification, version pinning. The canonical install instructions live at `https://specscore.md/install`, and the install skills are responsible for following them — init never reads that URL directly. Spec time should specify: the exact handoff protocol between init and install skills (input/output contract), how init resumes after install completes (re-detect PATH, re-invoke the failed step), and what happens when install is declined or fails (graceful degradation: continue with whatever bootstrap steps don't require the missing CLI, or abort cleanly with a clear next-step message). The handoff protocol is shared between both install skills — defined once, applied symmetrically.
- **`synchestra:install` skill existence.** This Idea presumes `synchestra:install` exists as a real skill (or will, by the time this Idea promotes to a Feature) with the same handoff contract as `specscore:install`. Confirm at specify time. If `synchestra:install` doesn't exist yet, scaffold a separate Idea for it (mirroring `specscore:install`) before specifying this one — do not absorb its scope into this Feature.
- **Conflict resolution when the snippet already exists at a different version.** Three-way merge? Replace-with-confirmation? Append-and-flag? Tied to the snippet-versioning rule in the [third-party-skill-integration-contracts](third-party-skill-integration-contracts.md) Idea. Spec time to pick one with a UX rationale once the snippet's versioning convention is settled.
- **Wizard question minimum vs. maximum.** What's the smallest set of questions that produces a sensible default init? What's the maximum before the wizard becomes annoying? Likely 3–5 questions in a single batch via `AskUserQuestion`, with sane defaults for everything else. Resolve at spec time.
- **`project.initialized` event payload.** What fields beyond `project_id`, `revision`, `cli_versions`? Confirm with the `synchestra-events.md` shared file. Likely a small set: artifact paths created, snippet-target file, CLI versions (or "agent fallback" markers), `change_summary`.
- **When does the delegation refactor in `ideate` and `specify` actually happen?** This Idea proposes it but doesn't block on it for MVP. Sequence: ship init → ship one round of dogfooding → refactor lazy-bootstrap in ideate/specify to delegate. Confirm sequencing at spec time and decide whether the refactor is a sub-Feature of init or its own Feature.
- **Where does the `synchestra init` CLI's contract live?** It's owned by the synchestra runtime repo, but this skill needs a stable contract to call against (flag set, exit codes, idempotence guarantees). Either the synchestra repo publishes a Feature for `synchestra init`, or this Feature documents the assumed contract and links to upstream when it stabilizes. Coordinate at spec time.
- **`specstudio:migrate` companion (forward reference).** Tracking item: when 3+ adopters request migration from non-SpecScore layouts (`docs/`-based, ad-hoc YAML front-matter, legacy `specscore-spec-repo.yaml`), scaffold the migrate Idea. Not blocking init.
